## 竞品比对（附录 A）

> 比对结论已融入下方 US/AC 设计，此处只列关键差异。完整对照表见末尾。

| 能力维度 | Dify | Coze | LangGraph Studio | n8n | **我们做不同** |
|---|---|---|---|---|---|
| 工具管理 | 内置插件市场，无自定义 Plugin/Skill/MCP | Skill=MCP+Prompt+Workflow 组合，Plugin 市场 | 无独立工具管理，工具在代码中定义 | 1300+ 集成节点，community nodes | **三种工具统一管理+各自 demo 模板+权限隔离** |
| 工作流编辑器 | DAG 拖拽+连线，智能布局，12+节点类型 | 可视化节点编排 | 代码定义+Mermaid 可视化，非拖拽 | Vue Flow 拖拽，AI 生成工作流 | **文本描述+Dify 式可视化双模式，工具快照隔离** |
| Agent 创建 | 对话流+Agent 节点 | Bot→Agent，选模型+绑 Skill | Python StateGraph 代码定义 | AI Agent 节点（LangChain） | **LangChain/LangGraph 自选底层+表单化选模型绑工作流** |
| Agent 编排 | 单 Agent 为主 | 单 Bot 为主 | 多 Agent 子图+条件路由 | 子工作流调用 | **Dify 式可视化 Agent 连线+并发/fork/一主多从** |
| 角色体系 | 无 | 无 | 无 | RBAC（编辑/只读） | **code-kit 12 角色体系可自定义+绑定到工作流节点** |
| 项目视角 | App 列表 | Bot 列表 | Thread 列表 | Workflow 列表 | **需求文档输入→自动解析→绑 Agent→端到端执行** |
| 调试/监控 | 轨迹回放+断点+实时日志 | 基础日志 | 时间旅行调试+状态检查点 | 每步数据预览+钉住数据 | **5 分钟粒度图表+审计日志+工具命中次数** |

## 用户故事

### M1 · 工具库 + 工作流

#### 模块 1 - 工具库

- **US-TOOL-1**：作为**开发者**，我想在界面上创建 Plugin，填写名称、描述、token 限制、可操作权限等属性，下载一个 demo 模板了解 Plugin 结构，以便快速开始开发自己的 Plugin。
- **US-TOOL-2**：作为**开发者**，我想在界面上创建 Skill，填写名称、描述、token 限制、可操作权限等属性，下载 demo 模板，以便理解 Skill 的组织方式。
- **US-TOOL-3**：作为**开发者**，我想在界面上创建 MCP，填写名称、描述、token 限制、可操作权限等属性，选择托管方式（Python 实现/外部服务），下载 demo 模板，以便避免 MCP 实现冲突。
- **US-TOOL-4**：作为**开发者**，我想用自然语言描述一个 MCP 接口（"我需要一个查询天气的 MCP"），系统自动生成 MCP 骨架代码和配置，以便零代码基础创建工具。
- **US-TOOL-5**：作为**开发者**，我想在工具列表中看到我创建的所有工具（Plugin/Skill/MCP），按类型筛选，查看详情，编辑和删除，以便管理我的工具库。
- **US-TOOL-6**：作为**admin**，我想看到所有用户创建的工具列表，可以查看、禁用、删除任何工具，以便全局管控。
- **US-TOOL-7**：作为**普通用户**，我只能看到和操作自己创建的工具，无法看到其他用户的工具，以保证数据隔离。

#### 模块 2 - 工作流市场

- **US-WF-1**：作为**开发者**，我想创建一个工作流，用文字描述使用哪些工具、调用顺序、约束条件、安全闸门、token 限制、每个阶段的 I/O 处理，以便声明式定义工作流。
- **US-WF-2**：作为**开发者**，我想在工作流编辑器中以 Dify 式可视化方式拖拽工具节点、连线端口、设置条件分支、并行执行，以便直观地组装复杂工作流。
- **US-WF-3**：作为**开发者**，我想在工作流中添加约束：调用流程（顺序/并行/条件）、安全闸门（前置校验/后置校验）、token 限制（单步/总量）、多角色策略（哪个节点需要哪个角色审批）、红线（不可跳过步骤），以便工作流具有完整的治理能力。
- **US-WF-4**：作为**开发者**，我想在工作流中设置多任务策略（任务拆分边界、并行度、自动/人工标记）、自动化策略（哪些步骤可由 AI 自动决策）、多角色对抗策略（审核节点），以便工作流能支撑复杂的 AI 开发流程。
- **US-WF-5**：作为**开发者**，当我将工具添加到工作流时，系统自动拷贝一份工具快照（「冻结」），后续我对工具原型的修改不影响已部署的工作流，以保证工作流的稳定性。
- **US-WF-6**：作为**开发者**，我想在工作流市场中查看我创建的所有工作流，按状态筛选（草稿/已发布/运行中），编辑和删除（运行中需先停止），以便管理工作流。
- **US-WF-7**：作为**admin**，我想看到所有用户的工作流列表，可以查看、禁用、删除任何工作流，以便全局管控。
- **US-WF-8**：作为**普通用户**，我只能看到和操作自己创建的工作流，无法看到其他用户的工作流，以保证数据隔离。

### M2 · Agent + 编排

#### 模块 5 - Agent 创建

- **US-AG-1**：作为**开发者**，我想创建一个 Agent，选择底层运行时（LangChain 或 LangGraph），填写模型 API 地址和 API Key（存储在环境变量/配置文件中），绑定一个已发布的工作流，设置 token 限制，以便 Agent 按工作流自动执行任务。
- **US-AG-2**：作为**开发者**，我创建 Agent 时绑定工作流后，系统自动解析工作流的结构（节点/顺序/约束/角色），在 Agent 详情中展示其能力说明，以便我确认 Agent 被正确赋予了预期能力。
- **US-AG-3**：作为**开发者**，我想在 Agent 列表中查看我创建的所有 Agent，看到每个 Agent 的状态（待命/运行中/错误）、绑定的工作流、模型信息、token 消耗总量，编辑和删除（运行中的 Agent 不可删除），以及被哪些项目引用，以便管理 Agent。
- **US-AG-4**：作为**admin**，我想看到所有用户的 Agent 列表，可以查看、禁用、删除任何 Agent，以便全局管控。
- **US-AG-5**：作为**普通用户**，我只能看到和操作自己创建的 Agent，无法看到其他用户的 Agent，以保证数据隔离。

#### 模块 6 - Agent 编排

- **US-AO-1**：作为**开发者**，我想用语言描述多个 Agent 的编排方式（"Agent A 先处理输入，输出喂给 Agent B 做分析，Agent C 审核 B 的结果"），指定并发/fork/一主多从/混合机制，系统生成编排配置，以便声明式定义多 Agent 协作。
- **US-AO-2**：作为**开发者**，我想在 Agent 编排画布上以 Dify 式可视化方式拖拽 Agent 节点、连线端口、设置数据流向、分支条件、聚合策略，以便直观地设计复杂的多 Agent 协作拓扑。
- **US-AO-3**：作为**开发者**，我想在 Agent 管理页查看 Agent 的更新历史、当前版本、被哪些项目引用，删除时如果在运行中则阻止并提示，以便安全管理 Agent 生命周期。
- **US-AO-4**：作为**开发者**，我想上传一个 YAML/JSON 配置文件（类似 K8s 资源定义），描述多个 Agent 的完整编排拓扑：每个 Agent 采用哪些工具组成工作流、SOP 调用顺序、Agent 间路由关系、并行关系、每个 Agent 使用的模型及配置、监控项，系统解析后自动生成可视化编排图并可直接执行，以便用 Infrastructure-as-Code 方式管理复杂的多 Agent 协作。

### M3 · 角色 + 组装 + 项目管理

#### 模块 3 - 角色工厂

- **US-RL-1**：作为**开发者**，我想创建一个自定义角色，参考 code-kit 内置的 12 角色模板（架构师/产品经理/安全审计师/等），定义角色的名称、性情描述、职责边界、触发场景、输入输出规范，以便在工作流节点中绑定使用。
- **US-RL-2**：作为**开发者**，我想查看 code-kit 内置角色的完整定义作为参考模板，在此基础上修改创建新角色，以便复用已验证的角色设计。
- **US-RL-3**：作为**admin**，我想看到所有用户创建的角色列表，可以查看、禁用、删除任何角色，以便全局管控。

#### 模块 4 - 角色×工作流×节点组装

- **US-CA-1**：作为**开发者**，我想将角色绑定到工作流的特定节点（如"安全审计师"绑定到审核节点），节点执行时自动调用对应角色的审查逻辑，以便工作流具有领域专家级别的审核能力。
- **US-CA-2**：作为**开发者**，我想将角色、工作流、节点三者任意组合——角色 A 负责工作流 X 的节点 1 和 2，角色 B 以对抗模式审核节点 3 的输出，以便构建复杂的多角色协作策略。

#### 模块 9 - 项目管理

- **US-PJ-1**：作为**开发者**，我想创建一个项目，填写项目名称和需求描述（可以是一句话、一段文字、一个接口文档 URL、或上传 markdown 需求文档），选择 Agent 和工作流，系统自动解析需求并给出执行方案，以便端到端启动 AI 开发任务。
- **US-PJ-2**：作为**开发者**，我想在项目列表中查看我创建的所有项目，看到项目状态（待开始/执行中/已完成/错误）、绑定的 Agent 和工作流、执行进度，编辑和删除（执行中不可删），以便管理项目。
- **US-PJ-3**：作为**开发者**，我想上传一个老旧项目（已有代码仓库），为其添加一个 Agent 进行代码分析、维护建议、新增功能，以便用 AI 能力维护遗留系统。
- **US-PJ-4**：作为**开发者**，需求文档无论包裹多少层（嵌套引用、多文件、接口文档中的子引用），系统总能递归总结，并给出执行方案。

#### 模块 10 - 项目监控

- **US-PM-1**：作为**开发者**，我想在项目监控页看到关联工作流的执行状态（当前节点、已执行时长、预计剩余时长），token 消耗趋势（5 分钟粒度柱状图/折线图/饼图），工具命中次数和按工具类型分布的饼图，使用频次趋势图，被哪些项目引用的表格，以及执行审计日志，以便实时掌握项目运行状况。
- **US-PM-2**：作为**开发者**，当工作流被赋予项目时，系统拷贝一份工作流快照，后续工作流原型的修改不影响该项目，以保证项目执行的一致性和可追溯性。

### 横切模块

#### 模块 7 - 安全策略

- **US-SEC-1**：作为**开发者**，我创建的任何实体（工具/工作流/Agent/项目）自动附带默认安全策略（闸门校验/I/O 过滤/token 软硬限制/权限检查）。**软限制**在执行中触发警告但不中断（保护已消耗 token 不浪费），**硬限制**在执行完成后阻断后续调用。我可以在高级设置中自定义覆盖，以便不遗漏安全配置的同时保留灵活性。
- **US-SEC-2**：作为**admin**，我想看到全局安全策略配置（token 上限、危险操作白名单、审计日志保留期），修改后对所有用户生效，以便统一管控。
- **US-SEC-3**：作为**开发者**，当我执行项目时，系统自动取项目→Agent→工作流→工具四层中最严格的 token 限制作为实际限制（`min(项目, Agent, 工作流, 工具)`），以便安全策略逐层收紧而非放宽。
- **US-SEC-4**：作为**开发者**，当一个 Agent 或工作流被项目引用时，不允许删除该 Agent/工作流，提示"被 N 个项目引用，请先解绑"，以防止运行中项目因上游资产删除而崩溃。

#### 模块 8 - 监控

- **US-MON-1**：作为**开发者**，我想在监控面板看到每个 Agent/工作流/工具的 token 消耗（按 5 分钟粒度的柱状图/折线图/饼图）、实际调用的模型名称、每个模型的 token 消耗分布（按 5 分钟粒度柱状图/折线图/饼图）、工具命中次数（按 5 分钟粒度的图表）、被哪些项目使用的列表表格、总体执行时间、平均执行时间、使用频次趋势（按 5 分钟粒度），以及执行审计日志，以便全面掌握资源消耗和模型使用情况。
- **US-MON-2**：作为**admin**，我想看到全局监控面板（所有用户汇总），按用户、实体类型、模型名称下钻，以便识别异常消耗和高成本模型。

#### 模块 11 - 完成标准

- **US-DONE-1**：作为**开发者**，前端应用能正常启动运行，所有页面可访问，所有接口在各种情况下经过严格测试（正常/异常/边界/并发），以便交付可用的产品。
- **US-DONE-2**：作为**开发者**，后端服务能正常启动运行，所有 API 端点经过严格测试（单元测试+集成测试+异常场景），以便交付可靠的服务。
- **US-DONE-3**：作为**使用者**，在本地部署后能通过浏览器访问完整功能，端到端执行真实任务（创建工具→组装工作流→创建 Agent→编排→绑定项目→执行→监控出图），以便实际演示和验证。

## 验收准则（AC）

### M1 · 工具库 + 工作流

#### AC-TOOL · 工具库

- **AC-TOOL-1**：**Given** 开发者已登录，**When** 点击「创建 Plugin」，填写名称/描述/token 限制/权限并保存，**Then** Plugin 出现在工具列表中，可下载 demo 模板 zip 包（含目录结构+代码骨架+README），US-TOOL-1 满足。
- **AC-TOOL-2**：**Given** 开发者已登录，**When** 点击「创建 Skill」，填写属性并保存，**Then** Skill 出现在工具列表中，可下载 demo 模板，US-TOOL-2 满足。
- **AC-TOOL-3**：**Given** 开发者已登录，**When** 点击「创建 MCP」，选择「Python 托管」并填写属性保存，**Then** MCP 出现在工具列表中，生成独立端口/进程隔离的骨架代码，可下载 demo 模板，US-TOOL-3 满足。
- **AC-TOOL-4**：**Given** 开发者输入「我需要一个查询天气的 MCP 接口，输入城市名，返回温度+湿度+天气描述」，**When** 点击「生成 MCP」，**Then** 系统生成含正确 JSON-RPC 接口定义的 Python 骨架代码、requirements.txt（mcp 依赖）、README，US-TOOL-4 满足。
- **AC-TOOL-5**：**Given** 用户 A 创建了 3 个 Plugin 和 2 个 Skill，**When** 登录后查看工具列表，**Then** 看到自己的 5 个工具，可按类型筛选；用户 B 登录看不到用户 A 的工具；admin 可以看到全部，US-TOOL-5/6/7 满足。
- **AC-TOOL-6**：**Given** 工具列表中某 MCP，**When** admin 点击「禁用」，**Then** 该 MCP 状态变为「已禁用」，所有引用该 MCP 的工作流在编辑时收到「工具已禁用」提示，US-TOOL-6 满足。

#### AC-WF · 工作流

- **AC-WF-1**：**Given** 开发者创建了一个包含 3 个工具的工作流（文本描述模式），填写了调用顺序、每个阶段的约束、安全闸门规则、token 限制、角色策略，**When** 保存并发布，**Then** 工作流出现在工作流列表中，详情页展示所有配置项，US-WF-1 满足。
- **AC-WF-2**：**Given** 工作流编辑器的可视化画布，**When** 开发者从左侧工具列表拖拽 3 个工具节点到画布、从输出端口连线到输入端口、设置条件分支节点（if/else）、添加并行执行标记，**Then** 画布正确渲染 DAG 图（无环约束自动检测），连接类型不兼容时红线提示，US-WF-2 满足。
- **AC-WF-3**：**Given** 工作流编辑器，**When** 开发者配置安全闸门（前置校验："输入必须包含 project_id 字段"）、token 限制（单节点 50k、总量 200k）、红线标记（「审核节点不可跳过」），**Then** 配置保存成功，工作流详情中展示「约束清单」卡片，US-WF-3 满足。
- **AC-WF-4**：**Given** 一个已发布的工作流被 3 个不同项目引用，**When** 开发者修改该工作流原型的工具配置（替换其中 1 个 Plugin），**Then** 已引用该工作流的 3 个项目的快照不变（继续使用旧版本），新项目引用时使用新版本，US-WF-5 的拷贝隔离满足。
- **AC-WF-5**：**Given** 工作流市场列表，**When** 用户 A 登录查看，**Then** 只看到自己创建的工作流；admin 看到全部；工作流类型标记（草稿/已发布/运行中），运行中的工作流删除按钮灰掉并提示「请先停止」，US-WF-6/7/8 满足。

### M2 · Agent + 编排

#### AC-AG · Agent

- **AC-AG-1**：**Given** 开发者创建 Agent，填写名称，选择「LangGraph」运行时，填写模型 API 地址（`https://api.openai.com/v1`）和 API Key（存入环境变量 `AGENT_<ID>_API_KEY`），从下拉列表选择一个已发布的工作流，设置 token 上限 500k，**When** 保存，**Then** Agent 出现在列表中，状态为「待命」，详情页展示：运行时类型、模型信息（API 地址脱敏展示）、绑定工作流名称+结构摘要、token 上限，US-AG-1/2 满足。
- **AC-AG-2**：**Given** Agent 状态为「运行中」（正在执行某项目的任务），**When** 开发者点击删除，**Then** 后端返回 400 "Agent 运行中不可删除"，前端提示「请先停止 Agent 后再删除」；非运行中的 Agent 删除成功，US-AG-3 满足。
- **AC-AG-3**：**Given** Agent 列表，**When** 查看，**Then** 每行展示：名称、运行时、绑定工作流、模型、状态（待命🟢/运行中🟡/错误🔴）、token 消耗总量、被引用项目数（可点击跳转），US-AG-3 满足。
- **AC-AG-4**：**Given** 用户 A 创建的 Agent，**When** 用户 B 登录查看 Agent 列表，**Then** 看不到用户 A 的 Agent；admin 看到全部，US-AG-4/5 满足。

#### AC-AO · Agent 编排

- **AC-AO-1**：**Given** 开发者输入编排描述「Agent A 先做代码审查 → Agent B 对 A 的结果做安全分析 → Agent C 汇总 A 和 B 的输出生成最终报告，A 和 B 可并行执行」，**When** 系统解析并生成编排图，**Then** 展示 A→B 串行、A→C 和 B→C 汇聚、A 和 B 标记为并行分支的可视化拓扑，US-AO-1 满足。
- **AC-AO-2**：**Given** Agent 编排画布，**When** 开发者拖入 3 个 Agent 节点，从 A 拖连线到 C 的输入端口，从 B 拖连线到 C 的输入端口，在 C 节点设置聚合策略「等待全部上游完成后合并」，**Then** 画布正确渲染，C 节点的输入端口显示两个上游连接，聚合策略保存成功，US-AO-2 满足。
- **AC-AO-3**：**Given** 开发者上传一份 YAML 配置文件，定义 3 个 Agent 的编排拓扑（Agent A 用工具组 W1 + 模型 gpt-4o → Agent B 用工具组 W2 + 模型 claude-sonnet → Agent C 汇聚 A 和 B 的输出），含路由关系、并行标记、SOP 调用顺序、监控项，**When** 系统解析并渲染，**Then** 可视化画布自动生成 3 个 Agent 节点+连线拓扑；YAML 语法错误时给出具体行号和修复建议；解析成功后可直接点击「执行」启动编排，US-AO-4 满足。

### M3 · 角色 + 组装 + 项目管理

#### AC-RL · 角色

- **AC-RL-1**：**Given** 角色创建页，**When** 开发者查看 code-kit 内置角色模板（如「安全审计师」），看到其性情描述/职责/触发场景/输入输出定义，在此基础上修改名称和职责后保存，**Then** 新角色出现在角色列表中，来源标记为「自定义（基于安全审计师）」，US-RL-1/2 满足。
- **AC-RL-2**：**Given** 角色列表，**When** 用户 A 查看，**Then** 看到 code-kit 内置 12 角色（只读）+ 自己创建的自定义角色；admin 看到所有内置+全部用户的自定义角色，US-RL-3 满足。

#### AC-CA · 组装

- **AC-CA-1**：**Given** 工作流编辑器，**When** 开发者为「代码审核」节点绑定自定义角色「我的代码审查员」，**Then** 节点详情展示绑定的角色名称和关键职责摘要，工作流执行到该节点时自动加载该角色的审查 prompt，US-CA-1 满足。

#### AC-PJ · 项目管理

- **AC-PJ-1**：**Given** 开发者创建项目，填写名称、上传 `需求文档.md`（内容包含嵌套引用：`详见接口文档 https://api.example.com/docs`），选择 Agent「代码生成 Agent」，**When** 保存，**Then** 系统解析主文档+递归抓取引用的接口文档，生成结构化需求摘要展示在项目详情中，US-PJ-1/4 满足。
- **AC-PJ-2**：**Given** 项目状态为「执行中」，**When** 开发者点击删除，**Then** 后端返回 400 "项目执行中不可删除"；非执行中项目删除成功，US-PJ-2 满足。
- **AC-PJ-3**：**Given** 开发者填写老旧项目 Git 仓库 URL，选择「代码维护 Agent」，**When** 创建项目，**Then** Agent 自动 clone 仓库、扫描代码结构、在项目详情中生成分析报告（技术栈/模块结构/代码质量问题），US-PJ-3 满足。

#### AC-PM · 项目监控

- **AC-PM-1**：**Given** 项目正在执行，**When** 开发者打开项目监控页，**Then** 看到：实际调用的模型名称、每个模型的 token 消耗趋势图（5 分钟粒度，柱状/折线/饼切换）、工具命中次数图（按工具类型分布的饼图+趋势折线）、使用频次趋势图、被哪些项目引用表格、总体/平均执行时间、审计日志列表（可按时间/操作类型筛选），US-PM-1 满足。
- **AC-PM-2**：**Given** 工作流 v1 被项目 P1 引用，开发者将工作流升级为 v2，**When** 查看项目 P1 的监控页，**Then** 绑定的工作流版本仍为 v1（快照），v2 不影响 P1 执行，US-PM-2 满足。

### 横切模块

#### AC-SEC · 安全策略

- **AC-SEC-1**：**Given** 开发者创建一个新工具/工作流/Agent/项目，**When** 不填写任何安全配置直接保存，**Then** 系统自动填充默认值：安全闸门=「无」（允许全部）、I/O 过滤=「不过滤」、token 软限制=「单次 80k/总量 800k」、token 硬限制=「单次 100k/总量 1M」、权限=「仅创建者可见」。软限制触发警告不中断，硬限制触发阻断。默认值可在高级设置中修改，US-SEC-1 满足。
- **AC-SEC-2**：**Given** 某工作流节点配置 token 软限制 40k、硬限制 50k，**When** 节点执行中 token 达到 40k，**Then** 系统发出警告通知（「token 已达软限制」）但**不中断执行**（保护已消耗的 token 不浪费）；**When** 该节点执行完毕且累计 token ≥ 50k，**Then** 硬限制生效——阻止该节点继续发起下一次调用，工作流暂停等待人工介入（降限或确认继续）；审计日志记录软限制触发时间+硬限制阻断时间。该行为不可通过前端绕过，US-SEC-1 满足（回应 G1 安全审计师建议）。
- **AC-SEC-3**：**Given** 工具硬限制 50k，工作流硬限制 200k，Agent 硬限制 500k，项目硬限制 300k，**When** 计算实际限制，**Then** 返回 `min(50k, 200k, 500k, 300k) = 50k`（工具最严，以工具为准）。若项目限制 100k 而 Agent 限制 500k，则返回 100k（项目更严），US-SEC-3 满足。
- **AC-SEC-4**：**Given** Agent A 被项目 P1 和 P2 引用，**When** 开发者尝试删除 Agent A，**Then** 后端返回 400 `"该 Agent 被 2 个项目引用，请先解绑"`；先解绑 P1 和 P2 后，再次删除 Agent A 返回 200，US-SEC-4 满足。

#### AC-MON · 监控

- **AC-MON-1**：**Given** admin 查看全局监控面板，**When** 选择时间范围「最近 1 小时」，**Then** 看到按用户分组的 token 消耗汇总、按实体类型（工具/工作流/Agent）分组的饼图、按模型名称分组的 token 消耗分布（柱状图/折线图/饼图可切换）、Top 10 消耗最高的实体表格，可点击下钻到单个实体详情，US-MON-1/2 满足。
- **AC-MON-2**：**Given** 某 Agent 绑定了模型 `gpt-4o` 并执行了 3 次任务，**When** 开发者打开该 Agent 的监控详情，**Then** 看到：实际调用的模型名称（`gpt-4o`）、该模型的 token 消耗趋势（5 分钟粒度柱状图/折线图/饼图，三种图表可切换）、按模型分组的 token 占比饼图。如果 Agent 曾切换过模型（如从 `gpt-4o` 切到 `claude-sonnet-5`），折线图按时间轴展示模型切换点和各模型的 token 消耗变化，US-MON-1 满足。

#### AC-DONE · 完成标准

- **AC-DONE-1**：**Given** 前端应用，**When** 运行 `npm run dev`，**Then** 所有页面可访问无白屏，所有接口调用覆盖正常返回/参数错误/未登录/token 过期/并发请求场景，每个页面的关键操作有自动化测试（vitest + React Testing Library），US-DONE-1 满足。
- **AC-DONE-2**：**Given** 后端服务，**When** 运行 `python main.py`，**Then** 所有 API 端点经过 pytest 测试覆盖（正常/异常参数/权限不足/未认证/并发），Swagger 文档（/docs）可访问，US-DONE-2 满足。
- **AC-DONE-3**：**Given** 本地部署环境（`ENCRYPTION_KEY` 已设、MySQL 已启动、Redis 已启动），**When** 访问 `http://127.0.0.1:<port>`，**Then** 能端到端完成：登录 → 创建 Plugin → 创建工作流（文本+可视化两种方式）→ 创建 Agent 绑定工作流 → 上传 K8s 风格 YAML 编排多 Agent → 创建项目绑定 Agent → 执行并查看监控图表出图，全链路无报错，US-DONE-3 满足。

## 非功能性需求

- **性能**：工具库列表加载 ≤ 2s（≤100 个工具）；工作流画布渲染 ≤ 3s（≤50 个节点）；DAG 无环检测 ≤ 500ms
- **安全**：前后端双重权限校验（延续现有体系）；API Key 加密存储于环境变量/MySQL，前端脱敏展示（`sk-***...***xYz`）；token 硬限制在服务端强制执行，不可通过前端绕过；审计日志自动记录所有 CRUD 操作和危险操作；MCP 托管独立进程/端口，避免与其他 MCP 冲突
- **可访问性**：亮色主题 WCAG 2.1 AA 对比度（≥ 4.5:1 正文）；暗色不做强制要求；工作流画布节点颜色不单独依赖颜色传递信息（同步标注文字状态）
- **兼容性**：桌面 Chrome/Firefox/Edge 最新 2 个版本
- **可观测性**：所有 tool/workflow/agent/project 操作记录到 audit log；token 消耗按 5 分钟聚合；错误日志输出到 stderr

## 依赖与假设

- **数据源**：工具/工作流/Agent/角色/项目元数据存储在 MySQL，与既有 `code-kit-monitor` 共用数据库实例
- **权限体系**：复用既有 `auth.py` 用户/角色/权限模型，新增实体（工具/工作流/Agent/项目）追加 owner_id + visibility 字段
- **监控图表**：复用前端已有 StatsTab/HealthTab 的图表组件（d3/recharts），扩展到新实体
- **假设**：M1 阶段 MCP 托管仅支持 Python，后续扩展其他语言
- **假设**：工作流可视化编辑器使用 React Flow（基于已有前端技术栈），不引入新框架
- **假设**：开发者有 Python 3.10+ 和 Node.js 18+ 运行环境
