# REQUIREMENT: 渠道 QR 码扫码接入

- **Change ID**: channel-qr-onboarding
- **关联**: `@.specs/channel-qr-onboarding/CHANGE.md`、`@.specs/CONTEXT.md`

---

## 用户故事

- **US-1**：作为**开发者**，我想在 Agent 渠道配置中点击「扫码接入飞书」并扫描弹出的二维码，以便**不需要手动查找和填写 App ID/App Secret**就能激活飞书渠道。
- **US-2**：作为**开发者**，我想在 Agent 渠道配置中点击「扫码接入钉钉」并扫描弹出的二维码，以便**不需要手动填写 Webhook URL 和签名密钥**就能激活钉钉渠道。
- **US-3**：作为**开发者**，我想在扫码流程失败、超时或二维码过期时看到**清晰的错误提示和重试入口**，以便知道下一步该做什么而不是面对一个静默的失败。
- **US-4**：作为**开发者**，我仍然可以**手动填写渠道凭证**来接入飞书/钉钉，以便在无法扫码时（无手机、服务器环境、未注册开放平台）仍有备选方案。
- **US-5**：作为**开发者**，我可以在本地开发时使用 **Mock 模式**模拟完整的扫码 OAuth 流程，以便不依赖真实飞书/钉钉平台账号也能调试和测试。

## 验收准则（AC）

每条用 Given / When / Then，必须可验证。

### US-1 · 飞书扫码接入

#### AC-1.1 · 发起扫码接入飞书

- **Given** 用户在 Agent 详情页，渠道配置区域
- **When** 用户点击「扫码接入」按钮并选择「飞书」
- **Then** 弹窗展示一个二维码图片 + 倒计时（5 分钟）+「请在飞书 App 中扫码」提示文字
- **验证方式**: 手动 UAT-1

#### AC-1.2 · 飞书扫码授权成功

- **Given** 二维码弹窗已展示，飞书开放平台已返回 device_code
- **When** 用户用飞书 App 扫码并确认授权
- **Then** 后端轮询检测到授权成功 → 弹窗自动关闭 → 渠道列表出现飞书渠道且状态为 `active`
- **验证方式**: 手动 UAT-1 / Mock 模式自动化

#### AC-1.3 · 飞书二维码过期

- **Given** 二维码弹窗已展示超过 5 分钟
- **When** 用户未扫码或飞书 device_code 已过期
- **Then** 弹窗展示「二维码已过期」提示 +「重新获取」按钮
- **验证方式**: 手动 UAT-1 / Mock 模式自动注入过期信号

#### AC-1.4 · 飞书用户拒绝授权

- **Given** 二维码弹窗已展示
- **When** 用户扫码后选择拒绝/取消授权
- **Then** 弹窗展示「授权已取消」提示 +「重试」和「手动填写」两个按钮
- **验证方式**: Mock 模式自动注入拒绝信号

### US-2 · 钉钉扫码接入

#### AC-2.1 · 发起扫码接入钉钉

- **Given** 用户在 Agent 详情页，渠道配置区域
- **When** 用户点击「扫码接入」按钮并选择「钉钉」
- **Then** 弹窗展示一个二维码图片 + 倒计时 +「请在钉钉 App 中扫码」提示文字
- **验证方式**: 手动 UAT-2

#### AC-2.2 · 钉钉扫码授权成功

- **Given** 二维码弹窗已展示，钉钉开放平台已返回扫码登录 URL
- **When** 用户用钉钉 App 扫码并确认授权
- **Then** 后端检测到授权成功 → 弹窗自动关闭 → 渠道列表出现钉钉渠道且状态为 `active`
- **验证方式**: 手动 UAT-2 / Mock 模式自动化

#### AC-2.3 · 钉钉扫码失败处理

- **Given** 钉钉扫码流程中发生网络错误或平台返回异常
- **When** 错误发生
- **Then** 弹窗展示具体错误信息（如「钉钉开放平台连接超时」）+「重试」按钮
- **验证方式**: Mock 模式自动注入网络错误

### US-3 · 错误处理

#### AC-3.1 · 轮询网络中断

- **Given** 二维码弹窗正在轮询授权状态
- **When** 网络中断导致轮询请求连续 3 次失败
- **Then** 弹窗展示「网络连接异常，正在重试…」并继续轮询，不关闭弹窗
- **验证方式**: Mock 模式自动注入网络中断

#### AC-3.2 · 平台 API 异常

- **Given** 后端在获取 device_code 或轮询时收到飞书/钉钉平台的非预期响应（5xx）
- **When** API 返回非预期状态码
- **Then** 弹窗展示「平台服务异常，请稍后重试（错误码: XXX）」+「重试」按钮
- **验证方式**: Mock 模式自动注入 5xx

### US-4 · 手动填表保留

#### AC-4.1 · 手动填表入口不受影响

- **Given** 用户在 Agent 详情页，渠道配置区域
- **When** 用户点击「添加渠道」（原有按钮）
- **Then** 原有的渠道类型下拉 + 凭证填写表单完整展示，功能与改动前完全一致
- **验证方式**: 手动 UAT-3 / 回归测试

#### AC-4.2 · 扫码和手动两种入口并存

- **Given** 渠道配置区域
- **When** 页面渲染完成
- **Then** 同时展示「添加渠道」（手动填表）和「扫码接入」（二维码）两个入口按钮，视觉上可区分
- **验证方式**: 手动 UAT-3

### US-5 · Mock 模式

#### AC-5.1 · Mock 模式完整流程

- **Given** 后端启动时 `CHANNEL_OAUTH_MOCK=true` 环境变量
- **When** 用户发起飞书/钉钉扫码接入
- **Then** 弹窗展示伪造的二维码（本地生成的 SVG 或占位图）→ 模拟 2 秒后自动「授权成功」→ 渠道创建且状态为 active
- **验证方式**: 自动化测试 / 本地启动带 `CHANNEL_OAUTH_MOCK=true` 手动验证

#### AC-5.2 · Mock 模式可注入错误场景

- **Given** Mock 模式启用
- **When** 前端传入 `?mock_scenario=expired` 或 `?mock_scenario=rejected` 参数
- **Then** 分别模拟二维码过期和用户拒绝授权两种场景
- **验证方式**: 自动化测试

## 范围切分

### v1（本次必做）

- 飞书 Device Code OAuth 扫码接入（真实 + Mock）
- 钉钉扫码 OAuth 接入（真实 + Mock）
- 手动填表入口完整保留
- 二维码弹窗 UI 组件
- 错误处理（过期/拒绝/网络异常/平台异常）
- Mock 模式（含场景注入：过期/拒绝/网络错误/平台异常）
- OAuth token/凭证加密存储（复用现有 encryption_service）

### v2（下一轮考虑，不本次）

- Slack OAuth 重定向接入（非二维码，但体验类似一键安装）
- 创建 Agent 成功后的「扫码接入飞书/钉钉」引导提示
- 二维码过期自动刷新（点击一次重新获取，不做自动）
- 扫码接入历史记录（哪个渠道、什么时间、成功/失败）

### out（永远不做）

- 微信/企微扫码接入（iLink Bot API 审批门槛高、流程差异大，需单独 change）
- SMTP/Telegram 扫码接入（这两个渠道没有 QR 码 OAuth 流程，保持手动填表即可）
- 在二维码中嵌入自定义品牌/Logo（复杂度远大于用户价值）

## 非功能性需求

- **性能**: 后端轮询飞书/钉钉 API 间隔 ≤ 5 秒；前端轮询后端间隔 ≤ 3 秒；全流程超时 5 分钟（与飞书 device_code 有效期一致）；二维码弹窗打开延迟 < 1 秒
- **可访问性**: 二维码图片附带 alt 文本描述；错误提示文字对比度满足 WCAG AA
- **安全**: OAuth state 参数防 CSRF（随机生成、单次有效、回调校验）；access_token/app_secret 加密存储（复用 encryption_service）；QR 码 URL 不拼接任何用户凭据；OAuth 流程关键步骤记录 audit log；轮询接口限流（同一 device_code 不超过 1 req/s）
- **兼容性**: 飞书开放平台 Device Code API v1；钉钉开放平台扫码登录 API v2；前端支持 Chrome/Firefox/Edge 最近 2 个主版本
- **可观测性**: OAuth 流程每个状态变更记录 audit log（device_code_created / user_scanned / authorized / rejected / expired / error）；MOCK 模式加 `mock: true` 标记

## 依赖与假设

- **依赖飞书开放平台**: Device Code API 可用性（https://open.feishu.cn/open-apis/auth/v3/app_access_token/ 等）
- **依赖钉钉开放平台**: 扫码登录 API 可用性（https://api.dingtalk.com/ 等）
- **假设**: 用户拥有飞书/钉钉管理员权限（扫码授权需要管理员身份）
- **假设**: 用户已在飞书/钉钉开放平台创建了应用（App ID/App Secret 由平台提供，扫码后自动填入）
- **假设**: 现有 encryption_service 和 channel_configs 模型足够承载新字段，无需数据迁移
