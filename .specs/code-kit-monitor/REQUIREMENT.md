# REQUIREMENT: code-kit 工作流监控面板

- **Change ID**: `code-kit-monitor`
- **关联**: `@.specs/code-kit-monitor/CHANGE.md`、`@.specs/CONTEXT.md`

---

## 用户故事

- **US-1**：作为**开发者**，我想在面板首页看到所有活跃 change 的当前阶段和进度，以便清窗回来后立刻知道「上次跑到哪了」。
- **US-2**：作为**开发者**，我想看到每个 change 的 task 执行状态（波次进度、自动化/人工、通过/失败），以便判断是否需要我介入。
- **US-3**：作为**开发者**，我想看到 6 个门禁的投票结果和每角色的投票理由，以便理解「为什么这个 task 被标了人工」。
- **US-4**：作为**开发者**，我想在面板上醒目看到中断任务，点击后能看到恢复指引（当前 task、下一步动作），以便快速续上。
- **US-5**：作为**开发者**，我想看到全链路 token 消耗的聚合数字，以便控制预算。
- **US-6**：作为**团队 lead**，我想看到所有成员的 change 列表和进展，以便识别哪些 change 卡住了。
- **US-7**：作为**使用者**，我想按 change-id / 关键词 / 状态搜索过滤，以便在海量 change 中快速定位。
- **US-8**：作为**开发者**，我想看到数据一致性校验的**告警列表**（缺了哪些产物、哪些 AC 没测试覆盖），以便快速定位问题并一键跳转修复。

## 验收准则（AC）

每条用 Given / When / Then，必须可验证。

### AC-1 · 工作流全景

- **Given** 仓库中存在 ≥1 个活跃 change（`.specs/<id>/` 目录存在且未被归档）
- **When** 打开监控面板首页
- **Then** 看到每个活跃 change 的卡片，含：change-id、当前阶段、进度（如「3/5 task 完成」）、状态（正常/阻塞/中断）
- **验证方式**: 手动创建 2 个模拟 change 目录，刷新面板，确认卡片数量和内容正确

### AC-2 · 中断任务醒目提示

- **Given** 存在一个 STATE.md 中「中断任务」字段非空的 change
- **When** 打开面板首页
- **Then** 该 change 卡片以醒目样式（红色边框/脉冲动画）展示在最顶部，点击展开后显示：当前 task-id、上次进度、恢复指令
- **验证方式**: 模拟写入 STATE.md 中断字段，确认面板置顶展示

### AC-3 · Task 执行状态

- **Given** 选中一个活跃 change 进入详情
- **When** 查看 Task 面板
- **Then** 看到波次列表，每个 task 显示：id、名称、`<auto>` 标记（🤖/👤）、状态（pending/in_progress/done/blocked）、重试次数；波次进度条显示当前 wave 完成比例
- **验证方式**: 解析 TASK.md 的 XML task 块，确认展示字段与文件内容一致

### AC-4 · 专家团门禁投票展示

- **Given** 选中一个活跃 change 进入详情
- **When** 查看「门禁」面板
- **Then** 看到 6 个 Gate（G1/G2/G2a/Task/G3/G4）的卡片，每个展示：通过/未通过/未触发状态、4 角色投票结果（✅/❌/⚪弃权）、每个角色的投票理由摘要
- **验证方式**: 模拟 CHANGE.md 末尾的 G1 投票记录段，确认面板正确解析并展示

### AC-5 · 产物只读查看

- **Given** 选中一个活跃 change 进入详情
- **When** 点击任意产物文件（CHANGE/REQUIREMENT/DESIGN/UI-DESIGN/TASK/SUMMARY/TEST/REVIEW）
- **Then** 以 markdown 渲染形式展示文件内容，**不提供编辑功能**；可以添加批注（纯前端存储，不写回文件）
- **验证方式**: 点击产物文件名，确认渲染正确，确认无编辑按钮，确认批注输入框可用

### AC-6 · Token 聚合展示

- **Given** 存在至少一份 `*-SUMMARY.md` 文件含 token 信息
- **When** 查看面板顶部状态栏
- **Then** 看到两个聚合数字：「本 change 已消耗 token」和「全仓库今日消耗 token」；不展示任何单次请求的 prompt/response 内容
- **验证方式**: 模拟 SUMMARY.md 中嵌入 token 记录，确认面板只展示聚合数字

### AC-7 · 搜索与过滤

- **Given** 仓库中存在 ≥5 个活跃 change
- **When** 在搜索框输入关键词或选择状态过滤
- **Then** change 列表实时过滤，支持按 change-id 模糊搜索、按阶段/状态筛选
- **验证方式**: 输入部分 change-id，确认只显示匹配项；切换状态过滤，确认列表更新

### AC-8 · Git 安全网检查点

- **Given** 当前仓库有 git history
- **When** 查看某个 change 的 Git 安全网面板
- **Then** 显示最近一次 safety commit 的位置（commit hash 前 7 位 + 时间）、从当前 HEAD 回滚到该 checkpoint 需要多少步
- **验证方式**: 手动创建 safety commit，确认面板正确识别并展示

### AC-9 · 数据一致性校验

- **Given** 选中一个活跃 change 进入详情
- **When** 查看「健康」面板
- **Then** 显示一致性校验结果**告警列表**：TASK task 数 vs SUMMARY 文件数是否匹配、REQUIREMENT AC 数 vs TEST 覆盖是否对齐、是否有缺失产物（如前端项目缺 UI-DESIGN）。每条告警可点击跳转到对应产物位置
- **验证方式**: 故意删掉一个 SUMMARY.md 制造不一致，确认面板标红提示且点击可跳转

### AC-10 · 安全事件醒目标记

- **Given** 仓库中存在 `.specs/` 下记录的秘钥扫描告警（如 TEST.md 安全轮记录了 trufflehog 命中）
- **When** 查看面板首页
- **Then** 该 change 卡片上出现红色安全告警图标，悬停展示告警摘要；**不展示**告警中的具体秘钥片段
- **验证方式**: 模拟 TEST.md 安全轮记录告警，确认面板展示告警图标

### AC-11 · 暗色/亮色双主题

- **Given** 打开监控面板
- **When** 首次加载或手动切换主题
- **Then** 默认暗色主题，可切换到亮色；亮色主题 WCAG AA 对比度通过 axe-core 检测（≥4.5:1 正文）
- **验证方式**: 用 axe-core 跑亮色主题关键页面，确认无 contrast violation

### AC-12 · 本地绑定

- **Given** 服务端启动
- **When** 从非本地地址（非 127.0.0.1 / localhost）发起 HTTP 请求
- **Then** 服务端拒绝连接或返回 403；只有本地请求能正常访问
- **验证方式**: `curl http://<内网IP>:<端口>/` 确认被拒绝，`curl http://127.0.0.1:<端口>/` 确认正常响应

### AC-13 · 门禁矩阵表（v1.1 · 来自专家评审）

- **Given** 选中一个活跃 change 进入详情 → 门禁 Tab
- **When** 查看门禁视图
- **Then** 以矩阵表形式展示：行=8 Gate（G1/需求门/G2/G2a/Task/G3/测试门/G4），列=参与角色，单元格=✅/❌/⚪ 投票结果。鼠标悬停显示理由摘要。不再使用折叠卡片
- **验证方式**: 打开 code-kit-monitor 的门禁 tab，确认矩阵表展示所有 gate × 角色

### AC-14 · 操作确认弹窗（v1.1 · 来自安全审计师）

- **Given** 用户在工作流编辑器/角色管理/文档编辑器中执行删除或修改操作
- **When** 点击删除按钮或保存按钮
- **Then** 弹出确认对话框：「此操作将修改底层配置/文件，确定继续？」用户确认后才执行
- **验证方式**: 删除一个角色，确认弹窗出现；点击取消，确认角色未被删除

### AC-15 · 缓存与扫描性能（v1.1 · 来自性能评审）

- **Given** 后端 scanner 已缓存扫描结果
- **When** 超过 5 秒后再次请求 /api/changes
- **Then** 缓存过期，重新扫描 .specs/ 目录；5 秒内重复请求返回缓存
- **验证方式**: 连续两次 curl /api/changes，中间 sleep 1，确认第二次更快（缓存命中）；sleep 6 后第三次请求，确认重新扫描

### AC-16 · 项目切换器（v1.1 · 来自用户体验官）

- **Given** 侧边栏底部有项目切换入口
- **When** 点击项目切换器
- **Then** 展示同级目录下所有含 .specs/ 或 code-kit/ 的项目列表，当前项目高亮。点击其他项目 → 切换 → 面板自动刷新数据
- **验证方式**: 点击项目切换器，确认列表中有多个项目，切换后 /api/changes 返回新项目数据

### AC-17 · 无障碍基础（v1.1 · 来自无障碍专家）

- **Given** 使用读屏器或键盘导航
- **When** Tab 键遍历侧边栏导航按钮
- **Then** 每个按钮有 aria-label（如「监控」「工作流」），折叠态图标按钮有文字描述；折叠动画在 prefers-reduced-motion 时关闭
- **验证方式**: 检查 HTML，确认每个导航按钮有 aria-label 属性；系统开 reduced-motion，确认折叠无动画

## 非功能性需求

- **性能**: 首屏加载 ≤ 3s（100 个 change 以内）；文件轮询间隔 5s
- **可访问性**: 亮色主题 WCAG 2.1 AA 对比度；暗色主题不做强制要求
- **安全**: 仅绑定 localhost（127.0.0.1），不暴露到网络；Token 数据仅聚合展示；不展示秘钥片段
- **兼容性**: 桌面 Chrome/Firefox/Edge 最新 2 个版本
- **可观测性**: 面板自身的错误日志输出到 stderr；前端 console error 有结构化格式

## 依赖与假设

- **数据源**: 依赖 `.specs/` 目录结构和产物文件格式（与当前 code-kit 版本绑定）
- **Git**: 依赖仓库有 git history 用于安全网检查点展示
- **技术约束**: 用户指定后端 Python + FastAPI、数据库 MySQL、缓存 Redis（如需）、向量数据库（如需，选最新版本）；正式技术栈选型走 2-design
- **假设**: `.specs/` 文件不会被外部进程并发修改（面板是唯一读取者）
- **假设**: 开发者有 Python 3.10+ 和 Node.js 18+ 运行环境

---

- **文件轮询**: 增量扫描（仅读 mtime 变更的文件），首次全量扫描后后续只读增量（避免 100+ change 场景 IO 抖动）

> AC 是 TEST 阶段派生用例的唯一来源，禁止在 TEST 阶段引入新 AC。
