# CHANGE: Agent 编排模块 k8s 化改造

- **Change ID**: `agent-k8s-orchestration`
- **创建时间**: 2026-07-04
- **状态**: 0-change（需求讨论中）

---

## Why

当前 Agent 编排模块（`OrchestrationPage`）支持手动拖拽 Agent 节点 + 连线定义拓扑关系（并发/fork/一主多从/混合），但存在以下问题：

1. **纯 UI 操作**，无可版本管理的声明式配置（无法 `apply -f`，无法 Git 管理拓扑变更历史）
2. **无运行时控制循环**，Agent 拓扑部署后处于"静态"——Agent A 挂了不会自动恢复，链路异常无自动降级/重试
3. **无可观测性**，编排拓扑运行时状态是黑盒，不知道哪个节点卡住、哪个链路慢
4. **无调度策略**，多个编排任务无优先级、无资源配额、无亲和性约束
5. **无模板化能力**，每次编排从零开始拖拽，无法复用已验证的拓扑模式

### 参考方案

借鉴 k8s 核心设计模式（声明式 API + 控制循环 + 调度器 + Operator + 可观测性三支柱），将 Agent 编排层从「静态连线图」升级为「自愈编排引擎」。详细映射关系见下方。

---

## What（本次做什么）

### A · 声明式配置

Agent 拓扑编排同时支持：

- **YAML/JSON 声明式配置**：用户编写/上传 YAML 描述 Agent 拓扑（节点定义、连线关系、并发策略、fork 条件、重试策略、优先级、token 配额），一键 `apply` 部署
- **UI 可视化编排**：保留现有拖拽+连线能力，但 UI 操作本质上是生成/修改 YAML，两者实时同步——拖拽改 YAML，YAML 改也更新画布
- **配置文件导入导出**：支持上传 `.yaml` / `.json` 部署，也支持下载当前拓扑为 YAML，纳入 Git 版本管理

### B · 控制循环（Reconcile Loop）

编排运行时增加持续监控能力：

- **observe → diff → reconcile**：后台 reconcile loop 持续比对编排期望状态（拓扑 YAML 中的定义）和实际状态（Agent 运行情况、连线状态、数据流）
- **自愈**：Agent 实例异常退出 → 自动按拓扑策略重启或降级到备用链路
- **自动重试**：链路节点返回异常/超时 → 按配置的重试策略（次数/退避/死信队列）自动重试
- **状态漂移检测**：手动改了运行中的 Agent → 检测到与期望不一致 → 告警 + 自动修复或提示人工介入

### C · 调度策略

多个编排任务并发时增加调度层：

- **优先级队列**：编排任务可设 priority（0-100），高优先级优先分配执行资源（Agent worker / token 配额）
- **资源配额**：每个编排任务声明 token 预算（软限制/硬限制），调度器按配额分配，防止一个任务吃光所有 token
- **亲和性/反亲和**：指定某些 Agent 必须在同一节点执行（亲和性），或必须分散在不同节点（反亲和性，容灾）
- **污点容忍**：执行节点标记「仅 GPU」「仅 CPU」，编排中的 Agent 声明自己能容忍哪些节点类型

### D · Operator 模式

编排模板化 + 一键实例化：

- **编排模板（类 CRD）**：把已验证的 Agent 拓扑保存为命名模板（如 `code-review-pipeline`、`doc-generate-pipeline`），含完整节点定义+连线+策略
- **模板参数化**：模板中变量用 `{{ .Values.xxx }}` 占位，实例化时填入具体值
- **一键部署**：选择模板 → 填参数 → `apply`，自动创建整套 Agent 拓扑
- **模板市场**：admin 可发布模板到公共模板库，其他用户浏览+一键复用

### E · 可观测性增强

编排拓扑运行时完全可视化：

- **拓扑实时状态图**：编排画布上每个节点实时颜色——绿色(健康)/红色(异常)/黄色(等待中)/灰色(未启动)，连线显示数据流方向+吞吐量
- **跨 Agent 调用链追踪**：一次编排任务的完整执行路径可追溯（Agent A → Agent B → Agent C，每段耗时/输入/输出/token）
- **拓扑级监控聚合**：每个编排实例的 token 消耗汇总、平均执行时间、被触发次数、成功率
- **审计日志**：编排拓扑的创建/修改/deploy/终止操作全部记录

### F · 最终一致性

编排拓扑的状态变更保证最终一致：

- **声明式更新**：修改拓扑 YAML（增删 Agent 节点/改连线/改策略）→ `apply` → 系统逐步将运行态收敛到新期望状态
- **渐进式收敛**：不中断正在执行的 Agent，新节点就绪后再切流，旧节点排空后移除
- **收敛状态可观测**：显示「期望副本 3 / 实际就绪 2 / 更新中 1」这类过渡状态

### 现有编排页改造

现有 `OrchestrationPage` 上的拖拽编排**平滑迁移**：
- 旧拓扑数据（`spec_json.nodes/edges`）自动导出为 YAML
- UI 保留拖拽能力，底层统一为 YAML 驱动（拖拽 = 实时修改 YAML）
- 旧拓扑无中断升级——迁移过程中已部署的编排实例继续运行

## 范围排除（本次不做）

- ❌ 不涉及单个 Agent 内部的工作流执行引擎改造（那是 AgentBuilder/WorkflowDetail 的职责）
- ❌ 不做真正的容器化/k8s 部署（不引入 Docker/Kubernetes 运行时），只借鉴 k8s 的**设计模式**，运行在本地 FastAPI 进程内
- ❌ 不做多租户集群调度（不跨机器、不涉及网络分区/跨节点通信）
- ❌ 不做 Helm Chart / Tanka 等 k8s 原生包管理工具的集成

---

## 验收线

1. 用户可编写 YAML 描述一套 Agent 拓扑，上传后一键部署，拓扑正确启动运行
2. UI 拓扑画布上的拖拽操作实时同步到 YAML 编辑器，反之亦然
3. 编排实例中任意 Agent 异常退出后 5 秒内自动重启或走备用链路
4. 拓扑实时状态面板正确展示每个节点的颜色（绿/红/黄/灰）和数据流方向
5. 编排模板可保存、参数化、复用、从模板一键部署
6. 调度器在多编排任务并发时按优先级+配额正确分配资源
7. 所有拓扑变更操作（创建/apply/删除）有审计日志
