- **Change ID**: `ai-dev-platform`
- **创建日期**: 2026-07-03
- **路径建议**: 完整（拆分 milestone）
- **状态**: draft

## Why（为什么做）

code-kit + code-kit-monitor 已完成工作流监控面板（v1.0~v1.3），覆盖了「看」的层面——看 change 进度、看 task 状态、看门禁投票、看 token 消耗。但开发者还需要「做」的层面：

1. **工具散落** — Plugin/Skill/MCP 各自为政，没有统一管理入口，创建靠手写文件
2. **工作流不可编排** — code-kit 的工作流是内置固定的，用户不能自己定义「先用 A skill 做 X，输出喂给 B plugin 做 Y」
3. **Agent 不可定制** — 不能创建自己的 Agent、选模型、赋予工作流、限制 token
4. **没有项目视角** — 每个 change 孤立运行，没有「这是一个需求文档，自动拆任务、执行、出结果」的端到端项目

本 change 将平台从**「监控面板」**升级为**「AI 开发平台」**——工具可管理、工作流可组装、Agent 可创建编排、项目可端到端执行。

## What（做什么）

在既有 code-kit-monitor（FastAPI + React/TypeScript + MySQL + 用户权限体系 + 审计日志 + 产物存储 + 监控面板）之上，新增以下能力：

### Milestone 1 · 工具库 + 工作流（基础能力）

| 模块 | 范围 |
|---|---|
| **1. 工具库** | Plugin / Skill / MCP 的创建、demo 模板下载、权限/token 限制、列表管理。MCP 支持语言描述创建。用户隔离（每人只看自己的），admin 全见 |
| **2. 工作流市场** | 工具任意组装为工作流。支持两种方式：① 文本描述（声明式）；② Dify 式可视化节点连线。工作流拷贝隔离（组装时工具快照，上游修改不影响已部署工作流）。含约束/安全闸门/token 限制/I/O 处理/多角色策略/红线/自动化策略 |

### Milestone 2 · Agent + 编排（执行能力）

| 模块 | 范围 |
|---|---|
| **5. Agent 创建** | LangChain/LangGraph 底层，选择模型（填 API/APIKey），绑定工作流，token 限制。Agent CRUD（更新/列表/被哪些项目引用/删除保护——运行中不可删） |
| **6. Agent 编排** | Agent 间编排：文本描述 + Dify 式可视化。支持并发/fork/一主多从等模式。Agent 管理 |

### Milestone 3 · 角色 + 组装 + 项目管理（深度能力）

| 模块 | 范围 |
|---|---|
| **3. 角色工厂** | 用户自定义角色（参考 code-kit 角色体系），含性情/职责/边界全要素 |
| **4. 角色×工作流×节点组装** | 角色绑定到工作流节点，任意组合 |
| **9. 项目管理** | 创建/删除/修改/列表/详情。研发类项目：一句话需求/接口文档/markdown 需求文档自动解析。绑定 Agent + 工作流。上传老旧项目添加 Agent 维护。删除保护（执行中不可删） |
| **10. 项目监控** | 通用工作流级监控（token 消耗/命中次数/执行时间/使用频次，5 分钟粒度柱/折/饼图 + 审计日志）。扩展既有 code-kit 监控为通用模式。工作流赋予用户时拷贝隔离 |

### 横切（贯穿所有 milestone）

| 模块 | 范围 |
|---|---|
| **7. 安全策略** | 每个实体（工具/工作流/Agent/项目）内置：安全闸门、I/O 处理、token 限制、权限策略。无则自动添加默认值 |
| **8. 监控** | Agent/工作流/工具的 token 消耗、命中次数、执行时间、使用频次（5 分钟粒度图表）+ 执行审计日志 |

## 视觉调性

- **选定**：工业（Industrial）— 沿用既有 `code-kit-monitor` 锁定调性
- **理由**：既有一致性，暗色默认、等宽字体、数据密度优先、冷色温
- **参考产品**：Grafana、Linear、VSCode

## 范围排除（这次不做）

- 不做公网部署/SaaS 化（延续 localhost 定位）
- 不做 Agent 运行时的高可用/集群调度
- 不做工作流市场的外部发布/分享/导入导出
- 不做自然语言生成完整工作流（仅模板+手动组装）
- 不做第三方工具市场的集成（如直接拉取 GitHub 上的 MCP server 列表）

## 验收线

- 用户能创建 Plugin/Skill/MCP，下载 demo 模板，设置权限和 token 限制
- 用户能用文本描述或可视化连线组装工具为工作流，工作流间拷贝隔离
- 用户能创建 Agent（选模型+绑工作流），Agent 能实际执行并产出结果
- 用户能创建项目（输入需求文档），绑定 Agent，端到端执行并查看监控

## 风险与未知

- MCP 协议版本兼容性：用户自建 MCP 可能与 code-kit 内置 MCP 冲突，需运行时隔离
- LangChain/LangGraph 版本锁定：需选定稳定版本，避免上游 breaking change
- 工作流拷贝隔离的性能开销：大工作流快照的存储和加载策略
- 竞品（Dify/Coze）在可视化工作流和 Agent 编排上已有成熟方案，需差异化定位

## 安全决策（G1 安全审计师 · 2026-07-03）

- **D1 · 引用保护**：Agent/工作流被项目引用时，**阻止删除**。返回 `400 "该 Agent 被 N 个项目引用，请先解绑"`。管理员需先进入每个引用项目解绑后才能删除。
- **D2 · Token 限制层级**：取**最严格值**。层级链：`min(项目限制, Agent限制, 工作流限制, 工具限制)`。项目设置了 500k，Agent 设置了 1M → 实际限制 500k（项目更严）。Agent 设置了 200k，项目设置了 500k → 实际限制 200k（Agent 更严）。

## 架构层影响声明

本 change 在既有 ARCHITECTURE（FastAPI 单体 + React SPA + MySQL）上扩展，不改变顶层模块结构。新增的 Agent 运行时和工作流引擎作为后端新模块接入，不修改既有 API 契约。ARCHITECTURE.md 不存在，本次 DESIGN 阶段将沉淀首批 ADR。

### 裁定 1 · 优先级排序

| 优先级 | 模块 | 理由 |
|---|---|---|
| **M1 先做** | 1.工具库 → 2.工作流 | 工具是原子单元，工作流是组合层。没有工具一切无从谈起。自底向上 |
